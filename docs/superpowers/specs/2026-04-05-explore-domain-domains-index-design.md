# explore-domain: domains.txt index — Design Spec

**Date:** 2026-04-05
**Status:** Approved

---

## Overview

`explore-domain` generates per-domain knowledge files but has no central registry of what domains have been documented. `read-knowledge` users can only match by slug (e.g. `src-auth`), not by meaning (e.g. "authentication"). This change adds `docs/conotate/domains.txt` — a plain-text index that `explore-domain` maintains and `read-knowledge` uses for richer matching.

---

## `docs/conotate/domains.txt` format

One line per domain:

```
<descriptive name>:<slug>
```

Example:

```
User authentication and session management:src-auth
UI component library:packages-ui-components
Payment processing pipeline:src-payments
```

- No header, no blank lines, append-ordered
- **Descriptive name**: 3–7 words, auto-generated by Claude during exploration — captures what the domain *does*, not just its path
- **Slug**: same slug used for the knowledge file (path with `/` → `-`)

---

## Changes to `explore-domain`

### Generation mode

After writing the knowledge file (current step 6), add:

**Step 7 — Generate description:**
Based on the exploration, write a short 3–7 word description of what the domain does (e.g. "User authentication and session management").

**Step 8 — Upsert `docs/conotate/domains.txt`:**
- File doesn't exist → create it with one line: `<description>:<slug>`
- File exists, no line ending with `:<slug>` → append `<description>:<slug>` as a new line
- File exists, line ending with `:<slug>` found → replace that line with `<description>:<slug>`

### Update mode

After updating the metadata header (current step 6), add:

**Step 7 — Refresh index entry (conditional):**
If the Overview section was rebuilt, regenerate the description from the new Overview and update the `:<slug>` line in `docs/conotate/domains.txt`.

---

## Changes to `read-knowledge`

### Domain Resolution (updated)

Replace the current slug-only resolution with a two-source lookup:

1. If `docs/conotate/domains.txt` exists: parse into `(description, slug)` pairs
2. Glob `docs/conotate/*.md` → derive slugs → add any not already present in domains.txt as `("", slug)` pairs (backward compatibility for files predating this feature)
3. Build a unified list of `(description, slug)`
4. For each user candidate (normalize: lowercase, `/` → `-`):
   - Match against slug (substring, case-insensitive)
   - Match against description (substring, case-insensitive, if non-empty)
5. **Unambiguous match** → proceed with that slug
6. **Ambiguous match** → numbered list, ask user to pick
7. **No match** → inform user, recommend running `explore-domain`

This means `auth` can resolve to `src-auth` via description match ("User **auth**entication..."), while `src-auth` still resolves via slug match.

---

## Relationship to existing design

This change adds a maintenance step to `explore-domain` and a richer lookup to `read-knowledge`. The knowledge file format (`<!-- domain: -->` blocks, domain-prefixed headings) is unchanged.
